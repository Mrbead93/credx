HOW TO LAUNCH
-------------
Open Termux and type:

  python3 ~/projects/credx/credx.py

You will see the CredX banner and the main menu.
Type a number and press Enter to select a module.
Press Enter after each module completes to return to the menu.
Type 0 to exit.


================================================================
  MODULE 1 — Breach Checker
================================================================

WHAT IT DOES:
  Checks if an email address appears in known data breaches
  using LeakCheck.io and breachdirectory.org. Also checks
  if a password has appeared in breach databases using
  k-anonymity — your password never leaves your phone.

HOW TO USE:
  Select option 1
  Enter an email address when prompted
  Then choose whether to check a password

EXAMPLE INPUT:
  Email address to check: youremail@gmail.com
  Check a password? (y/n): y
  Enter password: password123

EXAMPLE OUTPUT:
  [!] Found in 1 breach
  [BREACH] ClickASnap.com
  Date     : 2022-09
  Exposed  : Email addresses
  Verified : Yes

  [PWNED] Found 2,254,650 times in breach databases
  [!] Do not use this password anywhere

GOOD EMAILS TO TEST ON:
  Your own email address
  test@example.com (likely clean)

IMPORTANT NOTES:
  Password checking is completely free and private.
  Email checking uses free public breach APIs.
  A breach finding means change that password immediately
  on any site where you used it.


================================================================
  MODULE 2 — Default Credential Tester
================================================================

WHAT IT DOES:
  Scans a target for open services then tests 52 common
  default username and password combinations against them.
  Tests HTTP login forms, FTP and SSH services.

HOW TO USE:
  Select option 2
  Enter a target IP address when prompted

EXAMPLE INPUT:
  Target IP or hostname: 192.168.1.254

EXAMPLE OUTPUT:
  [OPEN] 80  — HTTP
  [OPEN] 443 — HTTPS
  Found 2 services — testing 52 credential pairs
  [SECURE] No default credentials found

  OR if credentials are found:
  [SUCCESS] admin:admin — http://192.168.1.1/admin

GOOD TARGETS TO TEST ON:
  192.168.1.254    (your home router)
  192.168.1.1      (alternative router IP)
  Any device on your own network

IMPORTANT:
  Only use against systems you own or have permission
  to test. Never run against systems you do not control.

CREDENTIAL PAIRS TESTED INCLUDE:
  admin:admin, admin:password, root:root,
  admin:1234, guest:guest, pi:raspberry,
  tomcat:tomcat, and 46 more combinations


================================================================
  MODULE 3 — Password Policy Analyser
================================================================

WHAT IT DOES:
  Analyses the security of a web login form by checking:
  - Whether login is served over HTTPS or plaintext HTTP
  - Whether CSRF tokens are present
  - Whether account lockout triggers after failed attempts
  - Whether rate limiting is in place
  - Whether password autocomplete is disabled

HOW TO USE:
  Select option 3
  Enter a target login URL when prompted

EXAMPLE INPUT:
  Target login URL: http://scanme.nmap.org

EXAMPLE OUTPUT:
  [CRITICAL] Login form not using HTTPS
             Credentials transmitted in plaintext

  [HIGH]     No CSRF token detected in form
             Login form may be vulnerable to CSRF

  [HIGH]     No account lockout policy
             Brute force attacks not mitigated

  [MEDIUM]   No rate limiting detected
             Login endpoint accepts rapid requests

  [LOW]      Password autocomplete not disabled
             Add autocomplete=off to password field

SEVERITY MEANINGS:
  CRITICAL = Immediate risk, fix urgently
  HIGH     = Serious vulnerability, fix soon
  MEDIUM   = Should be addressed
  LOW      = Minor improvement recommended

GOOD TARGETS TO TEST ON:
  http://scanme.nmap.org
  Any web login page on your own systems

IMPORTANT:
  This module sends multiple login attempts to test
  lockout policy. Only use on systems you own or
  have explicit permission to test.


================================================================
  MODULE 4 — Credential Exposure Scanner
================================================================

WHAT IT DOES:
  Searches multiple sources for exposed credentials
  and sensitive data associated with a domain:
  - GitHub code search for accidentally committed secrets
  - Pastebin for leaked credential dumps
  - Certificate transparency logs for email addresses
  - Direct file checks for exposed config files
  - Generates Google dork queries for manual searching

HOW TO USE:
  Select option 4
  Enter a target domain when prompted

EXAMPLE INPUT:
  Target domain: nmap.org

EXAMPLE OUTPUT:
  [*] Searching GitHub...
  [CLEAN] No exposed credentials found on GitHub

  [*] Checking for exposed files...
  [CLEAN] No sensitive files exposed

  [ Manual Google Dorks for nmap.org ]
  site:nmap.org filetype:env
  site:nmap.org filetype:sql
  site:nmap.org "api_key"
  "@nmap.org" password

  Exposure Summary:
  GitHub findings  : 0
  Pastebin hits    : 0
  Emails harvested : 0
  Exposed files    : 0
  Critical findings: 0

  OR if issues are found:
  [EXPOSED] config.py in username/repo
  [CRITICAL] Exposed: http://target.com/.env (1243b)

GOOD DOMAINS TO TEST ON:
  nmap.org             (well maintained, expect clean)
  Any domain you own

WHAT THE GOOGLE DORKS ARE FOR:
  Copy the generated dorks into Google manually.
  They search for exposed files, passwords and API keys
  associated with your target that Google has indexed.
  This is standard OSINT methodology used by pen testers.

FILES CHECKED FOR DIRECT EXPOSURE:
  .env, .git/config, wp-config.php, config.php,
  database.yml, settings.py, id_rsa, .htpasswd,
  backup.sql, passwords.txt and more


================================================================
  MODULE 5 — Full Credential Report
================================================================

WHAT IT DOES:
  Runs two modules automatically in sequence:
    Phase 1 — Breach checker on a specified email
    Phase 2 — Exposure scanner on the target domain
  Combines results into a single session.

HOW TO USE:
  Select option 5
  Enter a target domain when prompted
  Enter a primary email address to check

EXAMPLE INPUT:
  Target domain: example.com
  Primary email to check: admin@example.com

EXAMPLE OUTPUT:
  [*] Running breach check...
  [!] Found in 2 breaches — LinkedIn, Adobe

  [*] Running exposure scan...
  [EXPOSED] password.txt found at example.com/password.txt

WHEN TO USE THIS:
  When you want a combined credential intelligence picture
  of a target in one run. Good for starting an engagement
  or checking your own organisation's exposure.


================================================================
  RECOMMENDED WORKFLOW
================================================================

For assessing credential security of a target:

  STEP 1 — Module 4   Exposure Scanner
           Check what's already publicly exposed
           Input: target domain

  STEP 2 — Module 1   Breach Checker
           Check staff emails against breach databases
           Input: email addresses associated with target

  STEP 3 — Module 3   Policy Analyser
           Test the login form security
           Input: login URL of target web application

  STEP 4 — Module 2   Default Cred Tester
           Test for default credentials on open services
           Input: IP address of target

  OR — use Module 5 for a quick combined Phase 1 and 2.

For checking your own security:
  Run Module 1 on your personal email addresses
  Run Module 3 on any web applications you run
  Run Module 4 on your own domain


================================================================
  UNDERSTANDING YOUR FINDINGS
================================================================

BREACH CHECKER FINDINGS:
  Password in breach  = Change it everywhere immediately
  Email in breach     = Monitor for phishing attempts
  Paste exposure      = Email is circulating publicly

DEFAULT CRED FINDINGS:
  Credentials found   = Change default passwords now
  No credentials      = Good — default passwords changed

POLICY ANALYSER FINDINGS:
  HTTP login          = Upgrade to HTTPS immediately
  No CSRF token       = Developer needs to add protection
  No lockout          = Brute force is possible
  No rate limiting    = Automation attacks are possible

EXPOSURE SCANNER FINDINGS:
  GitHub exposure     = Revoke and rotate all keys found
  Exposed .env file   = Critical — take offline immediately
  Exposed .git        = Entire codebase may be accessible
  Emails harvested    = Use for phishing awareness training


================================================================
  LEGAL REMINDER
================================================================

Only run these tools against:
  Your own email addresses and accounts
  Your own web applications and servers
  Your own network devices
  TryHackMe lab machines (fully authorised)
  Any target with explicit written permission

The breach checker and exposure scanner use only
public data sources and are safe to run on any domain
for research purposes.

The default credential tester and policy analyser
send requests directly to target systems. Only use
these against systems you own or have permission to test.

Running credential testing tools against systems
you do not own is illegal under the Computer Misuse
Act 1990 and internationally.


================================================================
  SAVING AND VIEWING REPORTS
================================================================

Every module saves a timestamped report automatically to:
  ~/projects/credx/reports/

To list saved reports:
  ls ~/projects/credx/reports/

To read a report:
  cat ~/projects/credx/reports/FILENAME.txt

Report filename formats:
  breach_EMAIL_TIMESTAMP.txt
  default_creds_IP_TIMESTAMP.txt
  policy_URL_TIMESTAMP.txt
  exposure_DOMAIN_TIMESTAMP.txt
